01  Product Requirements Document (PRD)
1.1 Product Overview
VoiceScribe is a personal-use, web-based AI transcription platform designed for up to 20 trusted users. It processes audio and video content into structured, editable, and exportable transcripts, and applies AI post-processing to generate summaries, key insights, action items, chapters, blog posts, and social media content.
The platform supports direct file uploads (MP3, WAV, MP4, MOV etc.) and URL-based extraction (YouTube, direct media URLs), with the architecture designed to extend to Instagram, TikTok, and Facebook when platform regulations allow.
1.2 Problem Statement
Content creators, consultants, researchers, and educators regularly work with spoken-word media and need to convert it into actionable written content. Existing transcription tools are expensive, limited in exports, or locked into proprietary ecosystems. VoiceScribe solves this with a self-hostable, near-zero-cost, AI-powered workspace.

02  Technical Requirements Document (TRD)
2.1 System Architecture Overview
VoiceScribe uses a hybrid architecture separating concerns across four layers: the Next.js frontend/API on Vercel, the Supabase managed backend (database, auth, storage), a self-hosted Python microservice for transcription, and the Claude API for AI post-processing.
Architecture Diagram (Text)
[Browser]
    |
[Next.js on Vercel]  ←→  [Supabase: DB + Auth + Storage]
    |
[Whisper Microservice on VPS]  ←  yt-dlp / ffmpeg / faster-whisper
    |
[Claude API / Ollama]  ←  AI post-processing

2.2 Tech Stack Decisions
Layer	Technology	Reason
Frontend Framework	Next.js 14 (App Router)	SSR, API routes, Vercel-native
Language	TypeScript	Type safety, better DX
Styling	Tailwind CSS	Utility-first, rapid development
Components	Shadcn UI	Accessible, customisable, unstyled primitives
Icons	React Icons (Font Awesome)	Comprehensive, tree-shakeable

Transcription	faster-whisper	Fastest Whisper implementation
Media Extraction	yt-dlp	Supports 1000+ platforms
Audio Processing	ffmpeg	Universal audio/video conversion
AI Processing	Claude API (Anthropic)	Best quality at low volume
Deployment: App	Vercel Free Tier	Zero cost, auto-deploy

2.4 Backend API Routes
Route	Method	Description
/api/upload	POST	Receive file, store in Supabase Storage, create job
/api/jobs	GET	List jobs for current user
/api/jobs/[id]	GET	Get job status (for polling)
/api/transcribe	POST	Trigger Whisper microservice

/api/webhooks/transcription	POST	Receive completed transcript from microservice

2.5 Transcription Pipeline
5.	User uploads file or pastes URL
6.	Job created in DB (status: pending)
7.	File stored in Supabase Storage
8.	Next.js calls Whisper microservice asynchronously
9.	Microservice downloads media via signed URL
10.	If video: ffmpeg extracts audio track
11.	If URL: yt-dlp downloads audio, ffmpeg normalises
12.	faster-whisper processes audio → segments with timestamps
13.	Microservice POSTs results to Next.js webhook
14.	Next.js saves transcript to DB (status: completed)
15.	Original media deleted from Supabase Storage
16.	UI polling detects completion → renders transcript

2.8 Extensibility for Social Platforms
The media extraction layer uses a pluggable interface pattern. Adding Instagram, TikTok or Facebook support in future requires only creating a new extractor class — no core changes needed:
interface MediaExtractor {
  canHandle(url: string): boolean;
  extract(url: string): Promise<AudioBuffer>;
}
// Future: new InstagramExtractor(), new TikTokExtractor()

2.9 Security Strategy
•	All routes protected by Supabase JWT middleware
•	Row Level Security (RLS) on all DB tables — users see only their own data
•	Signed URLs for storage — no public media access
•	Webhook secret shared between Next.js and microservice
•	Rate limiting: max 10 uploads per hour per user
•	Input validation via Zod on all API inputs
•	yt-dlp sandboxed — args passed as array, never shell string
•	HTTPS enforced on all services
 
03  Application Flow
3.1 Visitor Flow
Land on voicescribe.app
  → Redirected to /login (no public pages)
  → Choose: Google OAuth  OR  Email + Password
  → Authenticated → Redirect to /dashboard

3.2 New User Onboarding
First login (profile.onboarded = false)
  → Welcome modal: 'Here's what VoiceScribe can do'
  → Step 1: Upload your first file or paste a URL
  → Step 2: Watch transcript generate in real time
  → Step 3: Try an AI feature (Summary or Blog Post)
  → Step 4: Export your transcript
  → Mark onboarded = true → Dismiss modal → /dashboard

3.3 Returning User Flow
Visit voicescribe.app
  → Middleware checks JWT → valid → /dashboard
  → See: recent transcripts, processing jobs, search bar
  → Click transcript → /transcripts/[id]
  → OR click 'New Transcript' → /upload

3.4 File Upload Flow
17.	User visits /upload
18.	Selects 'Upload File' tab
19.	Drag & drop or click to select file
20.	Client validates: file type and size (<500MB)
21.	If invalid: show error toast with specific message
22.	If valid: show file name, size, format preview
23.	Select options: language (auto-detect default), Whisper model
24.	Click 'Start Transcription'
25.	POST /api/upload — file streamed to Supabase Storage
26.	Job record created (status: pending)
27.	Microservice triggered asynchronously
28.	Redirect to /transcripts/[job_id] — processing state shown
29.	UI polls job status every 3 seconds
30.	On complete: transcript rendered
31.	On error: retry button shown with error message

3.5 URL Transcription Flow
32.	User visits /upload, selects 'URL' tab
33.	Pastes YouTube URL or direct media URL
34.	Client validates URL format
35.	Click 'Fetch & Transcribe'
36.	POST /api/transcribe/url
37.	yt-dlp validates, downloads audio
38.	Audio uploaded to Supabase Storage
39.	Whisper job triggered — same polling flow as file upload

3.6 Transcript Page Flow
The transcript detail page (/transcripts/[id]) has three tabs:
•	Tab 1 — Transcript: Read with timestamps, in-page search, edit button
•	Tab 2 — AI Insights: Summaries, key insights, action items, chapters
•	Tab 3 — Repurpose: Blog post, social posts, study/meeting/research notes
•	Sidebar: Export options (TXT, PDF, DOCX, SRT, VTT)

3.7 Export Flow
40.	User clicks Export in sidebar
41.	Export Modal opens
42.	Select format: TXT, PDF, DOCX, SRT, or VTT
43.	Optional: include timestamps, summaries, AI content
44.	Click Download
45.	POST /api/export/[id]?format=pdf
46.	Server generates and streams file to browser
47.	Toast: 'Download started'

3.8 Search Flow
48.	Type in dashboard search bar (min 2 characters)
49.	Debounced 300ms — query sent to /api/search
50.	Full-text search: titles + transcript content + summaries
51.	Results shown with title, date, duration, highlighted excerpt
52.	Click result → /transcripts/[id]

3.9 Error States
Error	Display Location	Message
File too large	Toast	File exceeds 500MB limit
Unsupported format	Toast	Format not supported. Use MP3, MP4, WAV...
Invalid URL	Inline	Please enter a valid YouTube or media URL
Download failure	Job status	URL could not be downloaded. Try file upload.
Transcription failure	Job status	Failed + Retry button
Export failure	Toast	Export failed. Please try again.
Session expired	Redirect	Session expired. Please sign in again.

05  Database Schema & Data Flow
5.1 Tables Overview
Table	Purpose	Key Fields
profiles	Extended user data (extends auth.users)	id, role, theme_preference, onboarded
transcripts	Core transcript records	id, user_id, title, source_type, status, duration
transcript_versions	Versioned transcript content	id, transcript_id, version_number, content, segments
ai_outputs	All AI-generated content	id, transcript_id, output_type, content
processing_jobs	Async job queue	id, transcript_id, job_type, status, attempts
exports	Generated export files	id, transcript_id, format, storage_path, expires_at

5.2 Core Table Schemas
transcripts
id              UUID PRIMARY KEY
user_id         UUID REFERENCES auth.users
title           TEXT DEFAULT 'Untitled Transcript'
source_type     TEXT CHECK IN ('upload', 'youtube', 'url')
source_url      TEXT
original_filename TEXT
duration_seconds INTEGER
language_detected TEXT
whisper_model   TEXT DEFAULT 'small'
status          TEXT CHECK IN ('pending', 'processing', 'completed', 'failed')
created_at      TIMESTAMPTZ DEFAULT NOW()

transcript_versions
id              UUID PRIMARY KEY
transcript_id   UUID REFERENCES transcripts
version_number  INTEGER
content         TEXT  — full transcript text
segments        JSONB — [{start, end, text}] array with timestamps
created_at      TIMESTAMPTZ DEFAULT NOW()

ai_outputs
id              UUID PRIMARY KEY
transcript_id   UUID REFERENCES transcripts
output_type     TEXT CHECK IN ('summary_short', 'summary_detailed',
                'key_insights', 'action_items', 'chapters',
                'blog_post', 'linkedin_post', 'twitter_thread',
                'instagram_caption', 'study_notes', 'meeting_notes')
content         TEXT
created_at      TIMESTAMPTZ DEFAULT NOW()

processing_jobs
id              UUID PRIMARY KEY
transcript_id   UUID REFERENCES transcripts
job_type        TEXT CHECK IN ('transcription', 'url_download', 'ai_processing')
status          TEXT CHECK IN ('queued', 'running', 'completed', 'failed', 'retrying')
attempts        INTEGER DEFAULT 0
max_attempts    INTEGER DEFAULT 3
error           TEXT
created_at      TIMESTAMPTZ DEFAULT NOW()

5.3 Table Relationships
auth.users
  └── profiles (1:1)
  └── transcripts (1:many)
      └── transcript_versions (1:many)
      └── ai_outputs (1:many)
      └── processing_jobs (1:many)
      └── exports (1:many)

5.4 Row Level Security (RLS)
RLS is enabled on every table. Users can only read and write their own data. Admin operations use the Supabase service role key on the server side only (never exposed to client).
-- Example policy
CREATE POLICY 'Users see own transcripts'
  ON transcripts FOR ALL
  USING (auth.uid() = user_id);

5.5 Storage Buckets
Bucket	Path Structure	Access	Retention
media-uploads	{user_id}/{job_id}/original.ext	Private (user-scoped)	Deleted after transcription
processed-audio	{user_id}/{job_id}/audio.wav	Private	Deleted after transcription
exports	{user_id}/{transcript_id}/{timestamp}.{format}	Private (user-scoped)	7 days

5.6 Full Data Flow
Upload → Transcript Flow
53.	Client POSTs file to /api/upload
54.	Server validates file type and size
55.	File uploaded to Supabase Storage (media-uploads bucket)
56.	INSERT into transcripts (status: pending)
57.	INSERT into processing_jobs (status: queued)
58.	POST to Whisper microservice with signed URL
59.	Microservice downloads, processes audio with faster-whisper
60.	Microservice POSTs results to /api/webhooks/transcription
61.	Webhook: INSERT transcript_versions (version 1, segments JSONB)
62.	Webhook: UPDATE transcripts (status: completed)
63.	Webhook: DELETE file from Supabase Storage
64.	Client SWR polling detects completed → renders transcript

AI Generation Flow
65.	User clicks 'Generate Summary' on transcript page
66.	Check ai_outputs for existing entry — return cached if found
67.	POST /api/ai/generate with { transcript_id, output_type }
68.	Server fetches transcript content from transcript_versions
69.	Builds prompt and calls Claude API
70.	INSERT result into ai_outputs
71.	Return content to client — rendered in AI panel
